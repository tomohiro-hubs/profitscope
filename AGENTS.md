# AGENTS.md

このドキュメントは、本リポジトリで作業する AI エージェント(Codex / Claude Code 等)および開発者向けのガイドラインです。
すべての実装・修正・追加機能は本ドキュメントのルールに従ってください。

---

## 1. プロジェクト概要

本プロジェクトは決算可視化アプリ「**ProfitScope**」です。
1年間分の勘定科目別データと売上高を入力すると、費用・各段階の利益・税引後利益を自動計算し、グラフと指標で可視化します。
ROI を含む経営指標を算出し、経営判断に使えるダッシュボードを提供することが目的です。

想定ユーザーは、中小企業経営者・個人事業主・経理担当者です。

---

## 2. 技術スタック

以下の技術構成を標準とします。明確な理由がない限り変更しないでください。

- フレームワーク: Next.js (App Router) + TypeScript
- スタイリング: Tailwind CSS
- チャート: Recharts
- 状態管理: React 標準機能(useState / useReducer / Context)
- バリデーション: zod
- データ保存: localStorage または JSON(初期段階)
- テスト: Vitest または Jest
- パッケージマネージャ: pnpm 推奨(npm でも可)

技術構成を変更する場合は `README.md` に変更理由を必ず記載してください。

## 3. ディレクトリ構成(推奨)

```
/app                  Next.js App Router のページ
/components           UI コンポーネント
/components/charts    チャート関連コンポーネント
/components/ui        汎用 UI 部品
/lib/accounting       会計計算ロジック(純粋関数)
/lib/finance          指標計算ロジック(ROI、利益率など)
/lib/format           通貨・数値フォーマット
/lib/validation       zod スキーマ
```

会計ロジック・指標計算ロジックは必ず `/lib` 配下に純粋関数として配置してください。

## 4. コーディング規約

### 4.1 言語・命名
- UI 文言、勘定科目名、コメント、README はすべて日本語。
- 変数名・関数名・型名は英語。意味の通る命名を行うこと。
- ドメインモデルは `interface`、ユーティリティ型は `type` を基本とする。

### 4.2 TypeScript
- `any` の使用を禁止。やむを得ない場合は理由をコメントで明記。
- 公開関数には必ず引数型と戻り値型を明示。
- 型は `/types` に集約し、使い回せる形で定義する。

### 4.3 関数設計
- 会計・指標計算は純粋関数として実装。副作用を含めない。
- 1 関数 1 責務を守る。
- 主要関数には日本語の JSDoc コメントを付ける。

### 4.4 UI
- ダッシュボード型 UI を採用し、要点が最初に目に入る構成にする。
- 表の羅列を避け、サマリーカード・グラフ・KPI を上部に配置。
- 各指標にはツールチップで簡単な説明を付ける。
- レスポンシブ対応(最低限デスクトップ・タブレット表示)。

### 4.5 数値・通貨表示
- 金額は `¥` 付き、3桁区切り、マイナス値は赤字で表示。
- 欠損値は `-` または `0` として扱い、UI 上で明示。
- 内部計算は整数(円単位)を基本とし、浮動小数点誤差を回避する。

## 5. ドメインルール(会計計算)

実装する損益計算の流れは以下を厳守してください。

1. 売上高
2. 売上原価
3. 売上総利益 = 売上高 − 売上原価
4. 販管費
5. 営業利益 = 売上総利益 − 販管費
6. 営業外収益 / 営業外費用
7. 経常利益 = 営業利益 + 営業外収益 − 営業外費用
8. 特別利益 / 特別損失
9. 税引前当期純利益 = 経常利益 + 特別利益 − 特別損失
10. 法人税等
11. 当期純利益 = 税引前当期純利益 − 法人税等

### ROI 計算
- ROI(%) = 利益 ÷ 投下資本 × 100
- 利益の定義(営業利益 / 経常利益 / 当期純利益)はユーザーが選択可能にすること。

### 主要指標
最低限、以下を実装すること。
- 売上総利益率
- 営業利益率
- 経常利益率
- 当期純利益率
- 費用率
- ROI

余力があれば損益分岐点売上高、月次推移、前年比較も実装する。

### 税金計算
- 初期実装では「法人税等」をユーザーが直接入力する方式を基本とする。
- 概算税率による自動試算はオプション機能として実装し、税率は設定値として外部化。
- 税率をハードコードしないこと。

## 6. データモデル(参考)

最低限、以下のような型を `/types` に定義してください。

```ts
export interface AccountItem {
  id: string;
  name: string;          // 勘定科目名(日本語)
  category: AccountCategory;
  amount: number;        // 円単位の整数
}

export type AccountCategory =
  | "revenue"            // 売上高
  | "cogs"               // 売上原価
  | "sga"                // 販管費
  | "nonOpIncome"        // 営業外収益
  | "nonOpExpense"       // 営業外費用
  | "extraordinaryGain"  // 特別利益
  | "extraordinaryLoss"  // 特別損失
  | "tax";               // 法人税等

export interface FinancialStatement {
  fiscalYear: number;
  items: AccountItem[];
  investedCapital: number; // 投下資本
}
```

## 7. サブエージェント編成と責務分担

本プロジェクトはサブエージェントを最大限活用し、**並列実装**を行います。
親エージェント(オーケストレーター)が以下の編成でサブエージェントを起動・統括してください。

### 7.1 オーケストレーター(親エージェント)の責務
- 全体アーキテクチャ・ディレクトリ構成・型定義・インターフェース契約の確定。
- 各サブエージェントへのタスク委譲(依存のないものは並列実行)。
- 成果物の結合、整合性検証、ビルド・テスト確認。
- コンフリクト・仕様齟齬発生時の裁定と再委譲。
- `docs/progress.md` による進行管理。

### 7.2 サブエージェント一覧

| # | エージェント名 | 担当領域 | 主な成果物 |
|---|---|---|---|
| 1 | Architect Agent | 全体設計・型定義・契約策定 | `/types`、`AGENTS.md`、`docs/architecture.md`、`docs/domain-model.md` |
| 2 | Accounting Logic Agent | 損益計算ロジック | `/lib/accounting` 配下の純粋関数群 |
| 3 | Finance Metrics Agent | 指標計算(ROI・各種利益率等) | `/lib/finance` 配下の純粋関数群 |
| 4 | Validation Agent | 入力検証 | `/lib/validation` の zod スキーマ |
| 5 | UI Component Agent | 汎用 UI 部品 | `/components/ui` 配下のコンポーネント |
| 6 | Chart Agent | 可視化チャート | `/components/charts` 配下のチャート |
| 7 | Dashboard Page Agent | 画面統合 | `/app` 配下のダッシュボード画面 |
| 8 | Sample Data Agent | サンプルデータ | `/data` 配下の年間/月次 JSON |
| 9 | Test Agent | ユニットテスト | `/tests` 配下のテストコード |
| 10 | Docs Agent | ドキュメント整備 | `README.md`、補足ドキュメント |
| 11 | QA / Integration Agent | 品質保証・統合検証 | ビルド・型・lint・テスト全通過 |

### 7.3 並列実行ルール
- 依存のないタスクは必ず並列実行する。
  - 例:Accounting Logic、Finance Metrics、Validation、UI Component、Chart、Sample Data、Docs は並列可能。
- 依存があるタスクは前段の成果物確定後に着手する。
  - 例:Dashboard Page Agent は他エージェントの成果物確定後、QA Agent は最後。
- 各サブエージェントは自分の担当領域以外のファイルを変更しない。変更が必要な場合は親エージェントに通知し、Architect Agent の調整を経ること。

### 7.4 サブエージェント間のインターフェース契約
- すべてのサブエージェントは Architect Agent が定義した型定義(`/types`)とインターフェースに従う。
- 契約変更が必要な場合は必ず親エージェントへ報告し、Architect Agent が更新したうえで全サブエージェントへ再周知する。

### 7.5 推論レベルの割り当て指針

各エージェントの作業特性に応じて、推論レベルを以下のように割り当ててください。
タスクごとに推論レベルを切り替えられる実行環境では、この割り当てに従うことでコストと品質のバランスが最適化されます。

| 推論レベル | 対象エージェント | 理由 |
|---|---|---|
| 非常に高い(Very High) | Architect Agent(初期設計フェーズのみ) | 全体方針・型・サブエージェント編成のブレが後工程に波及するため |
| 高(High) | Accounting Logic / Finance Metrics / Validation / Dashboard Page / QA / Integration | 数値の正確性、複数制約の同時充足、横断的な原因切り分けが必要 |
| 中(Medium) | UI Component / Chart / Sample Data / Docs | 仕様が明確で、標準的な実装で十分カバーできる |
| 低(Low) | 原則として使用しない | 会計アプリは数値の正しさが価値の中心であり、推論削減によるリスクが大きい |

運用上は、初期設計フェーズで一度だけ「非常に高い」を使い、それ以降は「高」を基本、UI / ドキュメント系のみ「中」に落とす二段構成を推奨します。

### 7.6 進行管理
- 親エージェントは各サブエージェントの進捗を `docs/progress.md` に一覧化する。
- 完了タスク・未完了タスク・失敗タスクを明示し、未完了/失敗は再委譲する。
- 最終的に QA / Integration Agent の検証が通った時点で全体完了とする。

- 会計計算ロジックと指標計算ロジックには必ずユニットテストを書く。
- 最低限、以下のケースを網羅すること。
  - 通常ケース(黒字)

  - 異常値(マイナス売上など)
- UI コンポーネントのテストは任意。
- テストは `pnpm test` または `npm test` で実行できる構成にする。

## 11. やってはいけないこと(禁止事項)

- 会計ルールを独自解釈で変更しない。「5. ドメインルール」に従うこと。
- 税率・勘定科目分類などの定数をコード中にハードコードしない。
- UI コンポーネント内で会計計算を行わない。
- `any` の濫用、型の省略、暗黙の型変換に依存した実装。
- 動作未確認のままコミットしない。
- 仕様に存在しない機能を勝手に大規模追加しない(小規模な改善は可、ただし README に明記)。
- ユーザー入力の検証をスキップしない。
- サブエージェントが自分の担当外のファイルを直接編集しない。

## 12. 実装作業の進め方

AI エージェントは以下の順序で作業してください。

1. リポジトリ構成と既存コードを確認する。
2. 仕様(`README.md` / 本ドキュメント / プロンプト)を再確認する。
3. Architect Agent が全体設計・型・契約を確定する(推論レベル:非常に高い)。
4. 依存のないサブエージェントを並列起動する(推論レベル:高〜中)。
5. 計算ロジックは `/lib` に純粋関数として実装し、ユニットテストを追加する。
6. UI コンポーネント・チャート・サンプルデータを並列で実装する。
7. Dashboard Page Agent が成果物を統合する。
8. QA / Integration Agent がビルド・型・lint・テストを全通過まで検証する。
9. Docs Agent が `README.md` と関連ドキュメントを最新化する。
10. 親エージェントが完了条件を確認し、`docs/progress.md` を最終更新する。

## 13. 完了条件(Definition of Done)

以下をすべて満たしたとき、タスクは完了とみなされます。

- アプリ「ProfitScope」がローカルで起動でき、サンプルデータで可視化画面が表示される。
- 入力編集に応じて指標とグラフがリアルタイム更新される。
- 売上高・総費用・売上総利益・営業利益・経常利益・税引前当期純利益・当期純利益・ROI が画面に表示される。
- 費用内訳グラフ、ウォーターフォールチャート、KPI 表示が実装されている。
- 会計ロジックが UI から分離され、純粋関数として独立し、ユニットテストが通る。
- ビルド・型チェック・lint・テストがエラーなく完走する。
- `README.md`、`AGENTS.md`、`docs/architecture.md`、`docs/domain-model.md`、`docs/progress.md` が最新化されている。
- サブエージェント並列展開で実装が行われ、各エージェントの担当範囲・成果物が `docs/progress.md` に記録されている。
